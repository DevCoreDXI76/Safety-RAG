# Safety-RAG 역할별 액션플랜 & 체크리스트

작성일: 2026-07-14
전제: 1인개발이 아니라 "1인사업"이라는 관점으로, 기획/디자인/개발/마케팅/CS/데이터분석/운영을 모두 본인이 겸임한다는 전제로 작성했습니다. 이전에 드린 [웹사이트전환·수익화 액션플랜]의 8주 로드맵을 "누가(어떤 역할이) 실행하는가" 기준으로 풀어낸 실행 문서입니다.

**사용법**: 모자를 바꿔 쓰듯 역할별로 시간을 블록화하세요. 한 번에 한 역할의 체크리스트만 붙잡고 끝내는 것이, 여러 역할을 동시에 왔다갔다 하는 것보다 진도가 빠릅니다. 각 항목은 `- [ ]`로 되어 있어 VS Code, Notion, GitHub 등에서 그대로 체크 가능합니다.

역할 순서는 실제 의존관계를 반영했습니다 — 기획이 방향을 정하고, 개발이 그 위에 인프라를 놓고, 디자인·데이터분석·운영이 붙고, 마지막에 마케팅·CS로 외부에 열립니다.

## 1. 🧭 기획자 (PM)

**범위**: 비즈니스모델 설계, 우선순위 결정, 베타 운영, 유저 리서치, 로드맵/백로그 관리. 다른 모든 역할의 "왜"를 결정하는 역할입니다.

### 상세 액션플랜

**A. 착수 전 — 방향 확정**
1. 목적 재확인 문서 1페이지 작성: "사이드프로젝트가 아니라 비즈니스"라는 전제를 스스로 서면화 (나중에 흔들릴 때 다시 읽을 용도)
2. 타깃 페르소나 구체화: 정보통신공사 안전관리 PM의 회사 규모, 연차, 하루 업무 패턴, 현재 쓰는 도구(엑셀? 한글?)를 최대한 구체적으로 적기
3. 경쟁 서비스(SafeYou 등) 최신 기능·가격 리서치 업데이트
4. BM 후보 2~3개 리스트업 (크레딧팩 / 개인구독 / 기업계약) — 장단점 비교표 작성

**A-1. 베타 0 실행 (1주차, 최우선 — 나머지 로드맵 전체의 게이트)**
5. 베타 0 참가자 3~5명 컨택 — 인맥 중 실제로 위험성평가표·TBM일지를 직접 작성해본 사람 우선
6. 베타 0 실행 — 텔레그램 봇 그대로 사용, "서식은 아직 없으니 텍스트를 복사해서 직접 써보고, 실제로 쓸 만한지, 시간이 줄어드는지"만 확인 요청
7. 베타 0 피드백 정리 — 4가지 확인: ① 초안 품질이 실사용 가능한가 ② 체감 시간 단축 여부 ③ 어떤 파일 형식이어야 실제 제출 가능한가(.hwpx vs 구버전 .hwp) ④ 지불 의향 금액
8. 베타 0 결과 기반 진행 여부 판단 — 통과 시 2주차부터 인프라 작업 계속 진행, 신호가 약하면 프롬프트/RAG 품질 개선으로 회귀 (개발자 역할과 공유하는 게이트 결정)

**B. 베타 1(정식 베타) 준비**
9. 베타 1 참가자 10~15명 리스트업 — 인맥별로 "이 사람에게 기대하는 역할"까지 메모 (후기? 실사용 피드백? 결제 의향 테스트?)
10. 인터뷰 질문지 작성: 사용 빈도, 지불 의향 금액, 필수 기능 vs 있으면 좋은 기능, 현재 서류 작성에 걸리는 시간, **"귀사는 `.hwpx`(신형 개방포맷)도 받으시나요, 구버전 `.hwp`만 받으시나요"** (개발자 역할 3-B 문서 내보내기 결정에 직결 — 베타 0에서 나온 답과 교차 확인)
11. 문서 생성 순서 가이드 콘텐츠 기획 (위험성평가표 → 작업계획서 → TBM일지, 프로젝트 변경 시 재생성 시점) — 인앱 가이드 백로그와 연결
12. 백로그를 Notion에 최신화 (기존 VSCode 스크립트 sync 활용)

**C. 베타 1 운영 및 상시**
13. 주간 지표 리뷰 루틴 확정 (요일·시간 고정)
14. 피드백 → 백로그 반영 프로세스 정의 (수집 → 분류 → 우선순위 재조정 주기)
15. Kmong 포트폴리오를 베타 결과로 업데이트

### ✅ 체크리스트
- [ ] 목적/비전 1페이지 문서화 완료
- [ ] 타깃 페르소나 정의 완료

- [ ] 베타 1 참가자 10~15명 리스트업 완료

## 2. 💻 개발자 (풀스택 + 인프라)

**범위**: API 마이그레이션, 인증·결제, 문서 내보내기(PDF/XLSX/HWPX), 배포·모니터링. 지금 백로그에서 가장 부피가 큰 역할입니다.

### 상세 액션플랜

**A. LLM 인프라 전환 (최우선 — 베타 0 시작 전 필수 선행조건)**
1. Anthropic Console에서 정식 API 키 발급 (Claude Pro 개인 계정과 완전 분리)
2. Railway Variables에 등록 — 로컬 `.env`와 별개 저장소라는 점 재확인 (과거 사고 재발 방지)
3. 코드 전체에서 구세대 모델 ID(`claude-3-5-*`) 검색 후 `claude-sonnet-5`, `claude-haiku-4-5-20251001`로 교체
4. 법정 별표/가이드라인 injection 블록에 `cache_control` 적용, TTL은 베타 기간 동안 1시간으로 명시
5. 토큰 사용량 로깅 테이블 구현 (`input_tokens`, `output_tokens`, `cache_read_input_tokens`, `cache_creation_input_tokens`, 문서유형, 타임스탬프)
6. 일일 비용 합산 후 임계치 초과 시 알림 로직 (텔레그램 봇으로 본인에게 알림도 방법)

**B. 신뢰성 확보**
7. 지게차(차량계 하역운반기계) 작업유형 코드 레벨 게이팅 — **베타 0 시작 전 필수 선행조건** (최소 조치): 미지원 시나리오는 생성 자체를 막고 "현재 지원 예정" 안내 반환
8. Haiku 다운그레이드 후보 문서유형(TBM일지, 안전보건교육일지)에 대해 Sonnet vs Haiku 결과를 나란히 저장하는 A/B 테스트 하네스 구축 (베타 0 게이트 통과 후, 2주차)
9. 최소 20~30건 A/B 테스트 실행, 기존 regex 인용 체커 통과율·수치 조작률 비교 (2주차)

**C. 인증·결제**
10. 회원가입/로그인 시스템 설계 (자체 이메일 인증 or 소셜 로그인 — B2B 신뢰도 고려 시 회사 이메일 인증 옵션 검토)
11. 세션/토큰 관리 구현
12. PG(포트원, 토스페이먼츠 등) 연동 — 결제 요청, webhook 검증, 실패/취소 처리
13. 크레딧/구독 상태 DB 스키마 설계
14. 요금제별 기능 제한·레이트리밋 로직 (기존 KST 자정 리셋 일일 제한 로직과 통합)

**D. 문서 내보내기 (PDF + XLSX + HWPX)**
15. **python-hwpx PoC (반나절, 최우선)** — 표준작업계획서 템플릿 1개를 `.hwpx`로 만들어 필드 채우기가 되는지 먼저 검증. `pyhwpx`는 Windows+한/글 설치가 필수라 Railway(Linux)에서 동작하지 않으므로 채택하지 않음 — 여기서 리스크를 먼저 제거
16. openpyxl 기반 XLSX 템플릿 바인딩 구현 (Node.js `SpreadJS` 대신 — 스택 통일, 라이선스 비용 없음)
17. HWPX 내보내기 구현 (python-hwpx)
18. PDF 내보내기 구현 (FastAPI 서버사이드, 범용 미리보기·제출용)

**E. 배포·운영**
19. dev/prod 환경변수 분리 (`API_BASE` 하드코딩 제거)
20. 에러 모니터링 도구(Sentry 등) 연동 검토
21. Railway Volume 백업 정책 점검

**F. 생성 속도(지연시간) 최적화** — 비용 최적화(A)와는 별개 과제. "30초+ 대기" 문제 해결용 (자세한 내용은 [Safety-RAG_웹사이트전환_수익화_액션플랜.md] 1-6절 참고)
22. 문서유형별 프롬프트에서 불필요한 서술형 부연설명 요구 제거 → 출력 토큰 길이 다이어트 (가장 저비용·최우선)
23. Haiku 전환 확정된 문서유형(TBM일지 등)에 실제 배포 — 비용뿐 아니라 처리량(속도) 개선 효과 겸용
24. 문서유형별 생성 시간(초) 로깅 추가 — 캐시 적중률과 함께 저장해 어떤 최적화가 실제로 효과 있었는지 사후 비교 가능하게 함
25. 스트리밍(SSE) 구현 — `messages.stream()` + FastAPI `StreamingResponse`, 프론트엔드는 `EventSource`/`getReader()`로 전환
26. 스트리밍 도입 전 결정: "완전 스트리밍(사후 검증 후 하이라이트)" vs "부분 버퍼링(섹션 단위 검증 후 순차 표시)" 중 택1 — 기존 regex 인용 체커와의 통합 방식이 여기서 갈림

## 3. 🎨 디자이너 (UI/UX)

**범위**: 스스로 인정한 약점 영역. "얼마나 직접 하고 얼마나 외주 줄지"를 먼저 정하는 것이 이 역할의 핵심 액션입니다.

### 상세 액션플랜

**A. 현재 자산 정리**
1. 기존 `index.html`의 CSS 변수(색상/폰트/여백 시스템)를 스타일가이드 문서로 정리 — 이미 톤앤매너가 잡혀 있어 처음부터 새로 만들 필요 없음
2. 신규로 필요한 화면 목록화: 회원가입/로그인, 요금제 선택, 결제, 크레딧/구독 상태 표시, 문서 히스토리 확장

**B. 와이어프레임**
3. 회원가입/로그인 화면 와이어프레임
4. 요금제/결제 화면 와이어프레임
5. 기존 결과 화면(마크다운 렌더링)의 가독성 개선 여부 검토 — 표/인용 블록이 많은 안전서류 특성상 여백·줄간격 재점검

**C. 외주 의사결정**
6. 직접 vs 외주 기준 정하기 (예: 예산 OO만원 이하 화면은 직접, 결제/온보딩처럼 전환율에 직결되는 화면은 외주)
7. (외주 시) 브리핑 문서 작성: 레퍼런스 서비스 3~5개, 톤앤보이스, 필수 화면 리스트, 기존 스타일가이드 첨부
8. (외주 시) 최소 2곳 이상 포트폴리오·견적 비교

**D. 실사용성 점검**
9. 모바일 화면에서 실제 사용성 테스트 — 타깃 유저가 "사무실 PC"보다 "현장에서 스마트폰"으로 쓸 가능성을 가정하고 점검
10. 폰트 크기/명도 대비 접근성 체크 (고령 사용자 포함 가능성 고려)

## 5. 🧾 운영·법무·재무 담당자

**범위**: 사업자등록, 약관, 결제 관련 법적 요건, 세무. **주의: 아래는 일반적으로 확인이 필요한 항목을 정리한 것이며, 저는 세무사·변호사 자격이 없습니다. 실제 등록·계약 전에는 반드시 세무사·법무사 등 전문가 확인을 받으세요.**

9. 최초 1회 전수 점검 완료 (현재 KB의 법령 인용이 최신인지)

## 8. 종합 — 8주 로드맵 대비 역할별 시간 배분 가이드

주 20시간 기준, 이전 액션플랜의 8주 타임라인에 맞춰 역할별로 시간을 어떻게 나눌지에 대한 가이드입니다. 정답은 아니고, 본인의 페이스에 맞춰 조정하세요.

| 주차 | 중심 역할 | 보조 역할 |
|---|---|---|
| 1주 | 개발자 (API/캐싱/지게차 게이팅) | 기획자 (방향 확정 + **베타 0 실행**) |
| 2주 | 기획자 (**베타 0 결과 리뷰 — 게이트**) | 개발자 (지게차 정식 구현, A/B 테스트) |
| 3주 | 개발자 (문서 내보내기: PDF/XLSX/HWPX) | 디자이너 (결과 화면 가독성) |
| 4주 | 개발자 (인증) | 디자이너 (로그인/가입 와이어프레임) |
| 5주 | 개발자 (결제) | 운영/법무 (약관, PG 심사서류) |
| 6주 | 데이터 분석가 (SDK 연동) | 디자이너 (요금제 화면) |
| 7주 | 기획자 + CS (**베타 1** 운영) | 마케터 (컨택 실행) |
| 8주 | 마케터 (확산) | 기획자 (피드백 반영) |

**원칙**: 개발이 몰려 있는 1~5주차엔 다른 역할은 "준비"만 해두고(문서 작성, 리스트업), 6주차 이후 실행 비중을 높이는 구조입니다. 모든 역할을 매주 동시에 굴리려 하면 오히려 아무것도 못 끝냅니다.
