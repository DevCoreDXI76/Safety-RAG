# Safety-RAG 도구형 웹사이트 전환 & 수익화 액션 플랜

작성일: 2026-07-14
대상: Safety-RAG (텔레그램 미니앱) → 독립 웹 도구로의 전환, CoreDXI의 첫 수익형 서비스

## 0. 총평 — 지금이 맞는 타이밍인 이유

Safety-RAG는 1인개발 원칙("사이드프로젝트가 아닌 비지니스", "유틸 서비스", "뾰족한 타깃")에 이미 가장 잘 부합하는 프로젝트입니다.

- **뾰족한 타깃**: 정보통신공사 안전관리 PM. 일반 안전관리 툴(SafeYou 등)이 못 건드리는 좁고 깊은 니치.
- **콜드스타트 문제 없음**: 유틸 서비스라 유저 1명만 있어도 가치가 발생. 플랫폼처럼 "사람이 없어서 탈퇴"하는 구조가 아님.
- **직접 판매 가능한 네트워크**: 20년간 쌓은 업계 인맥이 바로 첫 유료 고객 후보군.

지금 시점에서 "웹사이트로 빌드업"하겠다는 판단은 결과적으로 맞는 방향입니다. 다만 순서가 중요합니다 — **결제와 사용자 인증 없이는 아무리 잘 만들어도 매출이 0**이라는 점을 먼저 못박고, 아래 로드맵을 그 순서로 정렬했습니다.

### 0-1. 베타를 두 단계로 쪼개기 — 핵심 가설부터 먼저 검증

원래 8주 로드맵은 베타가 7주차에나 시작됩니다. 그 전에 인증·결제·문서 내보내기·분석 SDK를 6주 동안 다 만들어놓고 나서야 "이게 실사용 가능한가"를 물어보는 구조인데, 이건 순서가 거꾸로입니다. **핵심 가설(AI가 만든 초안이 실제로 쓸 만한가)이 검증되지 않은 상태에서 6주치 인프라부터 먼저 짓는 것**이라, 만약 베타에서 "내용이 별로다" 또는 "형식이 안 맞아서 못 쓴다"는 답이 나오면 그 6주는 상당 부분 헛수고가 됩니다. Daily Ritual Bot으로 먼저 완주 경험을 만드신 것과 같은 원칙이 여기도 적용돼야 합니다.

그래서 베타를 두 단계로 쪼갭니다.

**베타 0 (1주차, 즉시 시작 가능) — 콘텐츠 품질만 검증**
- 대상: 인맥 3~5명, 지금 있는 텔레그램 봇 그대로 사용 (웹 전환·결제·인증 전혀 필요 없음)
- 방법: "서식은 아직 없으니 텍스트를 복사해서 직접 써보시고, 내용이 실제로 쓸 만한지, 시간이 줄어드는지만 알려달라"
- 확인할 것 4가지: ① AI 초안이 실제로 쓸 만한가 ② 하루 2시간 걸리던 게 체감상 줄어드는가 ③ 어떤 파일 형식이어야 실제 제출 가능한가(.hwpx vs 구버전 .hwp) ④ 이 정도면 얼마를 낼 의향이 있는가
- 필요한 사전 작업은 딱 2가지뿐: **① 공식 API 전환** (외부인이 실사용하는 순간 Pro 계정 공유 리스크가 더 실질적으로 커지므로 필수), **② 지게차 작업유형 게이팅** (베타 테스터가 하필 이 시나리오를 테스트했다가 조작된 내용을 받으면 첫인상에서 신뢰를 잃을 수 있어 미리 차단)

**베타 1 (기존 7주차, 정식 베타)**
- 베타 0에서 검증된 가설 위에서, 웹 전환 + 문서 내보내기 + 결제까지 포함한 완전한 경험을 더 넓은 인원으로 테스트

베타 0은 나머지 로드맵 전체의 **게이트** 역할을 합니다. 결과가 좋으면 2~8주차 계획을 그대로 밀어붙이면 되고, "내용이 별로다"라는 신호가 강하면 나머지 인프라 작업을 잠시 멈추고 프롬프트/RAG 품질부터 다시 봐야 한다는 뜻입니다. 이 판단을 6주 늦게 하는 것보다 1주차에 하는 게 훨씬 쌉니다.

### 0-2. 경쟁 환경 재평가 — 타깃 세그먼트 재정의 (2026-07-15 업데이트)

베타 0 참가자들과의 대화에서 두 가지 중요한 신호가 나왔습니다.

1. **위존 '사파이어(SAPAIR)'** — 20년 이상 산업용 IT솔루션 기업이 만든 EHS(안전보건관리체계) 플랫폼으로, 울산 등 대기업 생산공장을 대상으로 이미 실제 계약을 진행 중입니다. 검색으로 확인한 바로는 사파이어의 실제 구매자는 "원청(공장 운영 대기업)"이며, 목표설정·사고기록·점검관리 등 경영시스템 레벨 기능이 핵심입니다 — "협력업체 현장 소장이 개별 위험성평가표를 즉시 뽑는" 실무 도구가 아닙니다. 다만 2026년 5월 "Safety AI Vision 2026" 세미나에서 PwC와 함께 Agentic AI 기반 콘텐츠 생성 기능을 시연한 바 있어, "AI가 초안을 써준다"는 차별점은 시간이 갈수록 옅어질 수 있습니다.
2. **포스코DX 사내 안전관리 플랫폼** — 개발자 본인이 재직 중인 회사가 실제 안전 문서·현장 사진 기반 벡터DB를 구축해 자체 플랫폼을 만들고 있다는 내부 정보가 확인됐습니다. 이는 순수 경쟁 문제를 넘어 **고용관계상 이해상충 리스크**이기도 해서, 겸업금지·직무발명 조항을 별도로 확인할 필요가 있습니다. 이 정보를 근거로 추가적인 내부 정보를 수집하는 것은 지양해야 합니다.

**결론 — 타깃 세그먼트를 다음과 같이 좁힙니다.**

| 세그먼트 | 판단 |
|---|---|
| 포스코 계열 협력사 | ❌ 제외 (이해상충 + 사내 플랫폼과 직접 경쟁) |
| 사파이어가 이미 계약한 대기업 생산공장 부속 협력사 | ❌ 제외 (원청 플랫폼 도입 시 개별 도구 도입 유인 감소) |
| 대기업 계열이 아닌 독립 중소 정보통신공사·재하도급업체·개인 소장/프리랜서 PM | ✅ 집중 — 애초에 사파이어 같은 원청 단위 엔터프라이즈 계약 대상이 아니고, 어떤 원청 플랫폼의 혜택도 못 받는 세그먼트 |

이에 따라 [Safety-RAG_영남권전략_리서치검토_및_업데이트.md]에서 핵심 GTM으로 짰던 "포항 포스코 협력사 네트워크"는 폐기하고, KICA 기술지도위원 채널도 "대기업 계열이 아닌 독립 업체" 위주로 좁혀 활용해야 합니다. 베타 0 질문지에도 "사파이어나 원청 자체 시스템을 써보셨는지, 커버 못 하는 부분이 무엇인지"를 추가해 이 재정의를 검증하세요.

## 1. LLM 비용/인프라 제안 검토 (요청하신 부분)

제안해주신 방향(공식 API 전환 + 하이브리드 모델 + 프롬프트 캐싱) 자체는 **모두 올바른 방향**입니다. 다만 세 가지를 짚고 넘어가야 합니다: **① 모델 세대가 바뀌었고, ② "RAG 검색"과 "생성"을 구분해야 하며, ③ 멀티벤더(OpenAI 혼용)는 지금 단계에선 득보다 실이 클 수 있습니다.**

### 1-1. 계정 공유 방식의 위험은 정확한 진단입니다

Claude Pro/구독형 계정은 **개인 사용을 전제로 한 요금제**이며, 이를 이용해 외부 다수 사용자에게 서비스를 제공하는 것은 계정 정지 리스크를 안고 가는 것이 맞습니다. 매출이 걸린 서비스에서 인프라가 통째로 사라질 수 있는 구조는 가장 먼저 없애야 할 리스크입니다. **이 항목이 전체 백로그에서 최우선인 것에 동의합니다.**

### 1-2. 모델명 업데이트 필요 — Claude 3.5 Sonnet/Haiku는 이전 세대입니다

2026년 7월 기준 현재 세대 모델과 API 가격은 다음과 같습니다 (100만 토큰당, 입력/출력):

| 모델 | API 모델 ID | 가격 (입력/출력) | 용도 |
|---|---|---|---|
| Claude Sonnet 5 | `claude-sonnet-5` | $2 / $10 (~8월 31일까지 도입가), 이후 $3 / $15 | 법령 해석, 인용 정확도가 중요한 생성 |
| Claude Haiku 4.5 | `claude-haiku-4-5-20251001` | $1 / $5 | 정형화된 표·일지 채우기 |
| Claude Opus 4.8 | `claude-opus-4-8` | $5 / $25 | Safety-RAG엔 과잉 스펙, 당장은 불필요 |

Sonnet 5의 도입가는 2026년 8월 31일까지만 적용되고 9월 1일부터 표준가($3/$15)로 전환되므로, 지금 파일럿·베타 단계를 이 도입가 구간에 맞추면 초기 비용 부담을 줄일 수 있습니다. Haiku 4.5는 100만 토큰당 입력 $1, 출력 $5로 가장 저렴한 현재 세대 모델입니다.

**실무 액션**: 코드베이스에 `claude-3-5-sonnet-*`, `claude-3-5-haiku-*` 같은 구세대 모델 ID가 하드코딩되어 있다면 지금 다 걷어내고, 위 표의 ID로 교체하세요. (참고로 3.5 세대는 이미 제공 종료 절차를 밟고 있는 모델도 있어, 방치하면 서비스가 갑자기 멈추는 사고로 이어질 수 있습니다.)

### 1-3. 정정: "RAG 검색"이 아니라 "생성" 단계에서 모델을 나눠야 합니다

RAG는 두 단계로 구성됩니다.

1. **검색(Retrieval)** — Voyage AI 임베딩으로 벡터 유사도 계산. **LLM 호출이 없고 비용도 거의 무시할 수준**입니다.
2. **생성(Generation)** — 검색된 컨텍스트 + 법정 별표 injection + 사용자 입력을 LLM에 넣어 최종 문서를 만드는 단계. **여기서만 Claude API 비용이 발생**합니다.

즉 "법률 해석이 필요한 RAG 검색 프로세스에 Sonnet"이라는 표현은 정확히는 **"생성 단계에서, 문서 유형별 난이도에 따라 모델을 분기"**로 이해하시면 됩니다. 현재 4종 문서를 기준으로 다음과 같이 나눠볼 것을 제안합니다.

| 문서 유형 | 특징 | 1차 제안 모델 |
|---|---|---|
| 위험성평가표 | 법정 별표 다수 인용, 수치 판단 필요 | Sonnet 5 |
| 표준 작업계획서 | 법조항 인용 + 현장 맞춤 서술 혼재 | Sonnet 5 |
| TBM일지 | 위험성평가표 요약 재구성, 정형 텍스트 위주 | Haiku 4.5 (테스트 후 결정) |
| 안전보건교육일지 | 법정 교육시간/내용 injection + 정형 텍스트 | Haiku 4.5 (테스트 후 결정) |

**중요한 단서**: 지금 겪고 계신 "환각이 카테고리를 옮겨다니는" 문제(법률 인용 형식 → 수치 조작)를 고려하면, **Haiku로 다운그레이드하기 전에 기존 regex 인용 체커가 Haiku 출력에서도 동일한 방어력을 보이는지 A/B 테스트가 반드시 선행되어야 합니다.** 저가 모델은 정확도가 떨어질수록 절감한 API 비용보다 재작업·신뢰도 손실 비용이 더 클 수 있습니다. TBM일지·교육일지부터 소규모로 전환해 최소 20~30건 정도 결과를 Sonnet 출력과 나란히 비교해보고, 인용 정확도·수치 조작률에 유의미한 차이가 없을 때만 전면 전환하세요.

### 1-4. GPT-4o-mini 혼용에 대한 재고 제안

GPT-4o-mini는 OpenAI의 이전 세대 저가 모델로, 지금 시점 기준으로도 세대가 지났을 가능성이 높습니다. 그보다 더 근본적인 문제는 **멀티벤더 자체의 비용**입니다.

- 별도 SDK/인증/에러 핸들링 유지보수 부담
- Anthropic 프롬프트 캐싱은 Claude API 안에서만 유효 — OpenAI로 분기하면 법정 별표 캐싱 이득을 그쪽에서는 다시 처음부터 설계해야 함
- 두 벤더의 응답 포맷·환각 패턴이 달라 지금 쌓아온 regex 검증 로직을 이원화해야 함

**제안**: 지금 단계에서는 **Claude 생태계 안에서 Sonnet 5 ↔ Haiku 4.5 이원화만으로 충분**합니다. Voyage AI 임베딩도 이미 Anthropic과 파트너십이 있는 임베딩 공급자라 스택이 자연스럽게 통일됩니다. 월 실사용 데이터가 쌓여 "이 정도 규모에서 벤더 분산이 실제로 유의미한 절감"이라는 근거가 나온 뒤에 멀티벤더를 재검토하세요. 지금은 복잡도를 늘릴 시점이 아닙니다.

### 1-5. 프롬프트 캐싱 — 정확히는 이렇게 작동합니다

Safety-RAG의 아키텍처(법정 별표·가이드라인을 프롬프트에 직접 injection)는 프롬프트 캐싱에 **거의 교과서적으로 들어맞는 케이스**입니다.

- 캐시 **쓰기**: 5분 TTL 기준 기본 입력가의 1.25배, 1시간 TTL은 2배
- 캐시 **읽기**: 기본 입력가의 0.1배 (즉 캐시가 적중하면 해당 부분 입력 비용이 최대 90% 절감)
- 최소 캐시 가능 길이: 대부분 모델 1,024토큰, Haiku 계열은 2,048토큰 — 법정 별표·가이드라인 텍스트는 보통 이보다 길어서 문제 없음
- 요청 하나에 캐시 breakpoint 최대 4개까지 지정 가능
- 기본 TTL이 2026년 3월부로 1시간 → **5분으로 변경**되었습니다. 명시적으로 `ttl`을 지정하지 않으면 조용히 5분으로 적용되니, 코드에서 `cache_control`에 TTL을 반드시 명시하세요.

**실무 설계 포인트**:
1. 프롬프트 구조를 `[고정: 법정 별표/가이드라인] → cache_control 브레이크포인트 → [가변: RAG 검색 결과 + 현장 정보]` 순서로 배치해야 캐시가 걸립니다. 순서가 조금이라도 바뀌면 캐시 미스입니다.
2. 문서 유형별로 injection되는 법정 텍스트가 다르므로, **문서 유형마다 캐시가 별도로 쌓입니다.** 초반 트래픽이 적을 때는 문서 유형별 요청 간격이 5분을 넘기기 쉬우므로, 베타 기간 동안은 **1시간 TTL**을 쓰는 편이 실질 적중률이 높습니다. 트래픽이 늘어 5분 내 재요청이 흔해지면 5분 TTL(쓰기 비용 저렴)로 전환을 검토하세요.
3. 캐시 적중률은 반드시 로그로 남기세요(`cache_read_input_tokens`, `cache_creation_input_tokens` 필드). 이게 곧 "실제 절감액"의 증거가 되고, 나중에 요금제 설계 시 원가 계산의 근거가 됩니다.

### 1-6. 생성 속도(지연시간) 최적화 — 비용과는 별개의 문제

원래 제안은 비용 최적화(하이브리드 모델 + 캐싱)만 다루고 있는데, 지금 겪고 계신 "생성에 30초 이상 걸린다"는 문제는 **비용과는 별개의 축**이라 따로 정리합니다.

**개념부터 분리해야 합니다.** "스트리밍"과 "실제로 빨라지는 것"은 다른 문제입니다. 스트리밍은 토큰이 만들어지는 대로 화면에 흘려보내는 것뿐이라 30초가 15초로 줄지는 않습니다. 대신 사용자가 빈 화면 대신 글자가 채워지는 걸 보게 되어 **체감** 대기시간이 줄어듭니다. 반면 아래 방법들은 **실제** 생성 시간 자체를 줄입니다.

**실제 생성 시간을 줄이는 방법 (우선순위 순)**

1. **프롬프트 캐싱 (1-5절, 이미 최우선 항목)** — 비용뿐 아니라 속도도 개선합니다. Anthropic이 공개한 벤치마크에서 10만 토큰급 긴 프롬프트 기준 캐시 적중 시 응답 시간이 11.5초 → 2.4초로 줄어든 사례가 있습니다. 법정 별표를 매번 새로 읽어들이지 않고 캐시에서 읽어오는 만큼 첫 토큰이 나오기까지의 시간(TTFT)이 줄어드는 원리입니다.
2. **모델 라우팅을 속도 관점에서도 재활용** — 1-3절에서 정한 Haiku 4.5 전환은 비용뿐 아니라 처리량(초당 생성 토큰 수) 자체가 Sonnet보다 빨라 실제 벽시계 시간도 줄입니다. TBM일지·교육일지처럼 A/B 테스트를 통과한 문서유형부터 적용하면 비용과 속도가 동시에 개선됩니다.
3. **출력 토큰 길이 다이어트 — 가장 직접적인 레버**: LLM은 토큰을 순차적으로 하나씩 생성하므로, 생성 시간은 거의 "출력 토큰 수 ÷ 초당 생성 속도"로 결정됩니다. 위험성평가표처럼 표가 긴 문서에서 프롬프트가 불필요한 서술형 부연설명을 요구하고 있지 않은지 점검하세요. 표 형태로 간결하게 채우도록 프롬프트를 다듬는 것만으로 출력 토큰이 줄고, 그만큼 생성 시간도 줄어듭니다. 코드 변경이 거의 없고 즉시 적용 가능해 **가장 먼저 시도할 항목**입니다.
4. **(선택, 리스크 있음) 문서를 섹션 단위로 쪼개서 병렬 호출**: 위험성평가표를 작업단계별로 나눠 여러 호출을 `asyncio.gather`로 동시에 쏘고 합치는 방식. 체감 단축 효과는 크지만 섹션 간 일관성(용어 통일, 앞뒤 맥락 참조)이 떨어질 수 있고, 캐시된 prefix가 호출마다 중복 적용되어 비용이 늘 수 있습니다. 품질 A/B 테스트 없이 바로 적용하기엔 위험해 **우선순위를 가장 낮게** 두세요.

**체감 대기시간을 줄이는 방법 (스트리밍 UX)**

Anthropic API의 `stream=True`와 FastAPI의 `StreamingResponse`를 SSE(Server-Sent Events)로 조합하는 것이 정석입니다.

```python
# FastAPI 엔드포인트 스케치
from fastapi.responses import StreamingResponse

async def generate_stream():
    async with client.messages.stream(
        model="claude-sonnet-5",
        max_tokens=4096,
        system=cached_system_prompt,  # cache_control 적용된 블록
        messages=[...],
    ) as stream:
        async for text in stream.text_stream:
            yield f"data: {text}\n\n"  # SSE 포맷

@app.post("/generate")
async def generate(req: GenerateRequest):
    return StreamingResponse(generate_stream(), media_type="text/event-stream")
```

지금 `index.html`의 `generateDraft()`는 `fetch` 후 `res.json()`으로 응답을 한 번에 받는 구조라, 프론트엔드도 `EventSource` 또는 `response.body.getReader()`로 스트림을 받아 `result-body`에 점진적으로 이어붙이는 방식으로 함께 바꿔야 합니다.

**설계 판단이 하나 필요합니다**: 지금 구축한 환각 방어 체계(regex 인용 체커)는 완성된 응답 전체를 검증하도록 되어 있을 텐데, 토큰을 실시간으로 흘려보내면 사용자가 검증 전의 "날것" 텍스트를 그대로 보게 됩니다. 두 방식 중 선택이 필요합니다.

- **완전 스트리밍**: 검증은 스트림 종료 후 사후에 돌리고, 문제 있는 인용/수치를 찾으면 결과 화면에서 해당 부분을 시각적으로 강조(노란 밑줄 + "확인 필요" 툴팁)하는 후처리 방식
- **부분 버퍼링**: 문단/섹션 단위로 모아 그 단위가 끝날 때마다 가벼운 체크를 돌리고, 통과한 섹션만 순차 표시 — 완전한 실시간은 아니지만 안전장치는 유지됨

안전서류라는 특성상 **부분 버퍼링 쪽이 더 안전한 기본값**이지만, 이는 품질/안전과 UX 사이의 트레이드오프라 최종 판단은 직접 하시는 게 맞습니다.

## 2. 재우선순위화된 백로그 (기존 백로그 + 수익화 요건 통합)

기존에 정리해두신 백로그(스트리밍 → 프롬프트 캐싱 → 템플릿+AI 하이브리드 → PDF/DOCX → 인앱 가이드 → 지게차 작업유형)를 **"매출로 이어지는가"** 기준으로 재정렬하면 다음과 같습니다.

| 순위 | 항목 | 이유 |
|---|---|---|
| 0 | **베타 0 실행** (인맥 3~5명, 텔레그램 봇 그대로) | 나머지 로드맵 전체의 게이트. 웹 전환·결제·인증 없이 1주차에 즉시 가능 (0-1절) |
| 1 | 공식 API 전환 + 프롬프트 캐싱 | 계정 정지 리스크 제거 + 비용 구조 확정 없이는 요금제 자체를 설계할 수 없음. **베타 0 시작 전 필수 선행조건** — 캐싱은 부수적으로 생성 속도도 개선(1-6절) |
| 2 | 사용자 인증 + 결제 연동 | 이게 없으면 나머지 모든 개발이 "매출 없는 개발"임 |
| 3 | 문서 내보내기: PDF + XLSX(openpyxl) + HWPX(python-hwpx) | 안전관리자는 결국 종이/문서로 제출·보관해야 함. 화면에 마크다운만 떠 있으면 **실사용 불가** 수준의 결핍임. (한글 자동화는 `pyhwpx`가 아닌 `python-hwpx`로 — 3-B절 참고) |
| 4 | 지게차(차량계 하역운반기계) 작업유형 정식 구현 | 미구현 시나리오에서 조작(fabrication)이 나는 상태로 유료 판매하면 첫 유료 고객에게 신뢰를 잃는 시나리오가 가장 위험. **베타 0 시작 전 필수 선행조건** |
| 5 | 생성 속도 1차 개선 (모델 라우팅 재활용 + 출력 토큰 다이어트) | 코드 변경이 작아 1번(캐싱)과 동시에 처리 가능. 스트리밍 없이도 체감 대기시간을 줄이는 가장 저렴한 방법 (1-6절) |
| 6 | 유저 이벤트 분석 SDK 연동 | Day1부터 붙여야 한다는 원칙을 스스로도 알고 계심 — 베타 오픈과 동시에 넣을 것 |
| 7 | 웹 UI 독립화 (텔레그램 종속 제거) | 3~6번이 끝나야 "팔 수 있는 상태"가 됨 |
| 8 | 인앱 가이드 | 리텐션/온보딩 개선, 결제 이후에도 늦지 않음 |
| 9 | 스트리밍(SSE) 구현 | 실제 생성 시간이 아니라 **체감** 대기시간을 줄이는 UX 레이어. 완전 스트리밍 vs 부분 버퍼링 여부부터 결정 필요 (1-6절) |
| 10 | 템플릿+AI 하이브리드 구조 개편 | 구조적 리팩터라 리스크가 크고 시간이 오래 걸림 — 유료 전환 이후, 데이터가 쌓인 뒤 진행 |

**변경 포인트 설명**: 원래 백로그에서 1순위였던 "스트리밍"을 뒤로 미룬 이유는, 스트리밍이 체감 UX는 개선하지만 유료 전환 여부와는 무관하고 실제 생성 시간을 줄이지도 않기 때문입니다. 대신 실제 시간을 줄이는 저비용 항목(모델 라우팅, 출력 길이 다이어트)을 5순위로 앞당겨, 스트리밍을 구현하기 전에 먼저 저렴하게 체감할 수 있는 속도 개선부터 확보하도록 했습니다. 문서 내보내기(PDF/XLSX/HWPX)와 결제 연동은 "이게 없으면 애초에 판매가 불가능한" 항목이라 여전히 앞쪽에 있습니다.

## 3. 인프라 마이그레이션 체크리스트

### 3-A. LLM API 인프라

1. **Anthropic Console에서 정식 API 키 발급** (Claude Pro 개인 계정과 완전히 분리)
2. **Railway Variables에 API 키 등록** — 기존에 겪으신 "로컬 `.env`와 Railway Variables는 별개 저장소"라는 교훈을 다시 한번: 여기서 실수하면 배포 후 API 호출이 전부 실패합니다.
3. **모델 ID 전면 교체**: `claude-sonnet-5`, `claude-haiku-4-5-20251001` 로 코드 내 하드코딩 검색·치환
4. **프롬프트 캐싱 적용**: 법정 별표/가이드라인 블록에 `cache_control` 명시, TTL 1시간으로 시작
5. **토큰 사용량 로깅**: 요청마다 `input_tokens`, `output_tokens`, `cache_read_input_tokens`, `cache_creation_input_tokens`를 DB에 저장 → 이게 향후 요금제 설계의 원가 근거가 됨
6. **비용 알림 설정**: Anthropic Console의 사용량 알림 또는 자체 일일 비용 합산 후 임계치 초과 시 알림(텔레그램 봇으로 본인에게 알림 보내는 것도 방법) — 예상치 못한 트래픽 급증으로 인한 과금 폭탄 방지

### 3-B. 문서 내보내기 아키텍처 (PDF + XLSX + HWPX)

B2B 고객(시공사 실무자)은 문서의 원본 서식 보존 여부에 민감합니다 — 화면의 마크다운만으로는 실사용이 안 되고, 국내 표준 양식(엑셀 표, 한글 문서)을 그대로 내려받을 수 있어야 신뢰를 얻습니다. 다만 이걸 구현하는 방법에서 한 가지 아키텍처 함정이 있어 바로잡습니다.

- **한글(.hwp) 자동화 — `pyhwpx`는 채택하지 않습니다.** `pyhwpx`는 Windows + 한/글(한컴오피스) 프로그램이 실제 설치된 환경에서만 동작하는 COM 자동화 라이브러리라, 지금 Railway의 Linux 컨테이너에서는 그대로 동작하지 않습니다.
- **대신 `python-hwpx`를 사용합니다.** 순수 파이썬 라이브러리로 한컴오피스 설치가 전혀 필요 없고, `.hwpx`(개방형 XML 기반 포맷 — 한국 행정기관의 표준 개방 문서 포맷이기도 함)를 직접 열고 표를 채우고 플레이스홀더를 치환할 수 있습니다. 기존 FastAPI(Python) 백엔드에 그대로 얹을 수 있어 별도 Windows 서버가 필요 없습니다.
- **엑셀(.xlsx) 자동화 — `SpreadJS`(Node.js, 유료)는 채택하지 않습니다.** 이미 파이썬 스택인 상황에서 Node.js 서비스를 새로 추가하고 라이선스 비용까지 낼 이유가 없습니다. **대신 `openpyxl`(파이썬, 무료)**로 위험성평가표의 [작업단계-유해위험요인-안전조치-위험도] 표를 정해진 템플릿 셀 좌표에 채워 넣는 요구사항을 동일하게 구현합니다.

**실행 순서 (리스크가 큰 것부터 먼저 검증)**

7. **python-hwpx PoC (반나절, 최우선)**: 실제 표준작업계획서 템플릿 1개를 `.hwpx`로 만들어 python-hwpx로 필드 채우기가 되는지 먼저 검증. 여기서 막히면 아래 항목들의 설계가 통째로 바뀌므로 다른 작업보다 먼저 리스크를 제거합니다.
8. **베타 인터뷰 질문 추가**: "귀사는 `.hwpx`도 받으시나요, 구버전 `.hwp`만 받으시나요"를 반드시 확인 (기획자 역할의 인터뷰 질문지에 추가). 정말 구버전 `.hwp`가 필수라는 응답이 많으면, 그때 Windows 미니PC/VM에서 pyhwpx를 돌리는 방식(비용·복잡도 증가)을 검토합니다 — 수요가 확인되기 전에는 투자하지 않습니다.
9. **openpyxl 기반 XLSX 템플릿 바인딩 구현**: 위험성평가표 등 표 구조가 복잡한 문서부터 우선 적용
10. **PDF 생성 구현**: 범용 미리보기·제출용 (FastAPI 서버사이드), HWPX/XLSX 대비 우선순위는 낮되 모든 문서유형에 공통 적용

## 4. 웹사이트 전환 시 확인이 필요한 지점 (현재 index.html 기준)

첨부해주신 `index.html`을 보면 이미 순수 웹 프론트엔드 구조(fetch 기반 SPA)로 짜여 있어 텔레그램 미니앱 SDK에 대한 직접 의존은 없어 보입니다. 다만 다음이 비어 있습니다.

- **인증 헤더가 전혀 없음**: 현재 코드는 `fetch(${API_BASE}/generate, ...)`를 인증 없이 호출합니다. 기존 백엔드가 HMAC 텔레그램 인증을 기대하고 있다면, 웹 단독 배포 시 이 인증 계층을 통째로 새로 설계해야 합니다. (옵션: 이메일/비밀번호 자체 회원가입, 또는 카카오/네이버 소셜 로그인 — 업계 특성상 개인 이메일보다 회사 도메인 이메일 인증이 B2B 신뢰도에 유리할 수 있음)
- **로컬 API_BASE 하드코딩**: `http://127.0.0.1:8000` — 배포 시 환경별 분기(dev/prod) 처리 필요
- **결제 연동 지점 없음**: 요금제/크레딧 개념이 UI에 전혀 없음 — 5절 참고

## 5. 수익 모델 설계

건설현장 안전관리 PM의 사용 패턴은 "매일 쓰는" 것이 아니라 **프로젝트 단위로 몰아서 여러 문서를 만드는 episodic 패턴**에 가깝습니다(위험성평가표 → 작업계획서 → TBM일지 순서로 이어지는 워크플로우를 이미 설계해두신 것도 이 패턴을 반영). 이 특성을 고려하면 다음 구조를 제안합니다.

| 티어 | 대상 | 구성 (예시) |
|---|---|---|
| 무료 체험 | 첫 방문자 | 워터마크 포함 초안 1~2건, PDF 내보내기 제한 |
| 개인/PM 구독 | 개인 안전관리 PM | 월 정액, 월 N건 포함, 초과 시 건당 과금 |
| 크레딧 팩 | 비정기 사용자 | 구독 대신 문서 건수 크레딧 선구매 (episodic 패턴에 더 잘 맞을 수 있음) |
| 팀/기업 | 시공사, 여러 현장 운영 | 다인 계정, 무제한 또는 대량 크레딧, 세금계산서 발행 |

**제안**: 처음부터 월 구독으로 확정 짓기보다, **베타 기간에는 크레딧 팩 vs 월 구독 중 어느 쪽을 실제 고객이 더 선호하는지 직접 물어보고 결정**하세요. 이건 정답이 없는 영역이고, 20년 인맥 안의 실제 PM 몇 명에게 "한 달에 몇 건 만드는지, 프로젝트당 얼마면 쓸 만한지"를 먼저 물어보는 게 가장 빠른 검증입니다.

**결제 연동**: 국내 개인/소규모 사업자가 붙이기 수월한 PG로 포트원(PortOne, 구 아임포트), 토스페이먼츠 등이 일반적으로 거론됩니다. 정확한 수수료·심사 요건·세금계산서 발행 가능 여부는 시점에 따라 달라지므로 계약 전 각 사 홈페이지에서 최신 조건을 직접 확인하시길 권합니다. (참고: 저는 재무/법률 자문 자격이 없는 만큼, 사업자등록·전자상거래업 신고·세금계산서 발행 요건은 세무사·법무사 확인을 권합니다.)

## 6. 마케팅 실행 계획 (직접 쓰신 전자책 원칙 반영)

전자책에서 강조하신 "직접 몸으로 부딪히는 마케팅", "10~15명 직접 베타"를 그대로 적용하면:

1. **베타 0 (1주차, 이미 별도 게이트로 진행 — 0-1절 참고)**: 인맥 3~5명, 텔레그램 봇 그대로 콘텐츠 품질만 검증. 이 결과가 좋아야 아래 단계로 넘어갑니다.
2. **베타 1 = 정식 베타 (7주차, 2주)**: 업계 인맥 10~15명에게 직접 연락 → 웹 전환된 버전으로 3~5건 생성해보게 하고 피드백 수집. 이 단계 목표는 매출이 아니라 **"환각 없이 실사용 가능한가"** 검증.
3. **확산 (베타 1 피드백 반영 후)**: 디스콰이엇에 정식 소개 글(이전에 폴잇으로 해보신 포맷 재사용), 블라인드 "직접 홍보" 채널에 정보통신공사/건설안전 태그로 게시.
4. **카카오톡 오픈채팅방**: 정보통신공사·전기안전 관련 오픈채팅방이 실제로 존재합니다. 직접 검색해서 참여 후 가벼운 소개(스팸처럼 보이지 않게 커뮤니티 룰 확인 후).
5. **네이버 블로그**: 건설안전·산업안전 카테고리 블로거에게 협업 제안 — 이전 폴잇 사례처럼 리뷰 콘텐츠 요청 (원고료 밴드도 비슷하게 적용 가능).

**중요**: 이 서비스는 소개팅앱과 달리 **B2B 전문직 툴**이라 인스타그램 광고보다는 업계 커뮤니티·인맥 기반 채널의 전환율이 훨씬 높을 가능성이 큽니다. 광고비를 태우기 전에 무료 채널부터 소진하세요.

### 6-1. 콘텐츠/SEO 채널 (2026-07-15 추가 — 베타 이후 Phase 2)

**포지셔닝을 명확히 해야 합니다**: 안전 관련 콘텐츠(재해사례, 법령 해설, 체크리스트 등)를 무료로 계속 쌓는 아이디어는 **사파이어·포스코DX 같은 경쟁자에 대한 차별화 무기가 아닙니다.** 그건 엔터프라이즈 관계·내부 데이터 우위의 싸움이라 콘텐츠로 이길 수 없습니다. 대신 이건 0-2절에서 재정의한 타깃(대기업 계열이 아닌 독립 중소 시공사·프리랜서 PM)에게 닿는 **인바운드 채널**로 목적을 잡아야 합니다 — "위험성평가표 작성법" 같은 검색으로 들어오는 사람들은 KICA 인맥 채널로는 못 닿는 사람들입니다.

- **수익화**: 애드센스 등 광고 기반으로 크레딧/구독료 할인 재원을 만든다는 아이디어는 지금 단계에서는 보류를 권합니다. 한국어 B2B 콘텐츠는 광고 단가가 낮아 신생 사이트 트래픽으로는 유의미한 할인 재원이 되기 어렵고, 유료 법적 서류 도구 페이지에 배너 광고가 붙으면 "KOSHA 조문 인용 정확도로 전문성을 증명한다"는 차별화 메시지와 오히려 상충됩니다. 광고보다는 무료 도구(예: 법정 안전교육 시간 자동 계산기)로 이메일을 수집해 무료체험 전환으로 연결하는 **리드 제네레이션** 방식을 추천합니다.
- **재해사례 콘텐츠**: AI로 사례를 새로 생성하지 말고, KOSHA·언론이 이미 공식 발표한 사례를 **출처를 명시해 요약·큐레이션**하는 방식으로 한정하세요. 실제 사망·중상 사고를 잘못 기술하면 법적 문제로 번질 수 있고, 저작권 문제도 있습니다.
- **타이밍**: 베타 0/1로 핵심 제품이 검증되기 전에는 시작하지 않습니다. 지금 20시간/주가 이미 개발+베타+관계형 마케팅으로 꽉 차 있어서, 콘텐츠 제작이라는 세 번째 상시 업무를 얹는 건 MVP 우선 원칙에 어긋납니다. 8주 로드맵 이후, 베타 1 결과가 좋을 때 백로그에 추가하세요.

## 9. 8주 타임라인 (주 20시간 기준)

| 주차 | 작업 |
|---|---|
| 1주 | 공식 API 전환, 모델 ID 교체, 프롬프트 캐싱 적용, 토큰 로깅, 출력 길이 다이어트, 지게차 작업유형 게이팅(최소 조치) → **베타 0 실행** (인맥 3~5명, 텔레그램 봇 그대로, 웹 전환·결제·인증 불필요) |
| 2주 | **베타 0 결과 리뷰 — 다음 단계 진행 여부 판단(게이트)**. 통과 시: 지게차 작업유형 정식 구현 + 회귀 테스트(기존 4종 문서 포함 재검증) + python-hwpx PoC(반나절, 리스크 제거용) |
| 3주 | 문서 내보내기 구현: PDF + XLSX(openpyxl) + HWPX(python-hwpx) |
| 4주 | 사용자 인증(회원가입/로그인) 설계 및 구현 |
| 5주 | 결제 연동(PG 선정 및 붙이기), 요금제 UI |
| 6주 | 유저 이벤트 SDK 연동, 무료 체험 플로우 QA |
| 7주 | **베타 1(정식 베타)** — 인맥 10~15명 대상, 웹 전환 포함 완전한 경험 |
