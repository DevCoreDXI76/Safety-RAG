# Safety-RAG — PM을 위한 Agent 운영 구조 가이드

작성일: 2026-07-14
목적: 당신(PM)이 각 역할이 정확히 어떻게 동작하는지 이해하고, 판단·승인만으로 프로젝트를 굴릴 수 있도록.

---

## 0. 먼저 정직하게: 이건 "완전 자율 멀티에이전트"가 아닙니다

"각 분야별 Agent가 알아서 일하고 나는 결재만" — 이 그림 자체는 정확히 구현됩니다. 다만 오해하시면 안 되는 게 하나 있습니다: **Agent들이 서로 대화하며 저절로 진행되는 건 아닙니다.** 매번 "이 항목 진행해줘"라고 트리거를 걸어주는 사람은 여전히 PM인 당신입니다. 대신 그 이후 — 리서치하고, 초안 쓰고, 코드 짜고, 검증하는 — 실제 노동은 전부 Agent 쪽으로 넘어갑니다. 당신에게 남는 건 "이 결과물 괜찮은가, 다음으로 넘어가도 되는가"라는 판단뿐입니다. 이게 사실상 원하시는 것과 거의 같은 결과입니다.

---

## 1. 세 실행 주체가 정확히 무엇을 하는가

| | Claude Code | Claude Projects (×5) | Claude Cowork |
|---|---|---|---|
| **담당 역할** | 개발자 (24개 항목) | 기획자·디자이너·데이터분석가·운영법무재무·마케터·CS | 여러 역할 중 서브태스크가 몰린 작업 |
| **입력(당신이 주는 것)** | 체크리스트 항목 1개 + 관련 코드/문서 경로 | "다음 항목 진행해줘" 한 마디 (Project가 이미 역할·맥락을 알고 있음) | 하나의 큰 목표 ("마케팅 채널 4개 세팅해줘") |
| **처리(Agent가 하는 것)** | 실제로 파일을 읽고, 코드를 고치고, 테스트를 돌림 | 그 역할 관점에서 리서치·초안·비교표 작성 | 스스로 계획을 세우고 여러 서브태스크로 쪼개서 순차 처리 |
| **출력(당신이 받는 것)** | 코드 diff, 실행 결과, 통과/실패 여부 | 문서 초안, 비교표, 체크리스트 결과 | 완료된 여러 산출물 묶음 |
| **PM이 확인할 것** | 이 코드가 의도대로 동작하는가 | 이 판단·초안이 실제로 쓸 만한가 | 전체 산출물이 방향에 맞는가 |
| **승인 게이트** | 코드 반영 전 항상 diff를 보여줌 (자동 내장) | 당신이 "좋아, 다음" 이라고 말해야 다음으로 넘어감 | 세션 끝나면 결과 전체를 검토 |

**핵심 차이 하나만 기억하시면 됩니다**: Claude Code는 "손"이고, Claude Projects는 "역할별 전문가"이고, Cowork는 "여러 일을 한 번에 처리하는 대행자"입니다.

## 2. 실제 사이클 예시 — 항목 하나가 시작부터 끝까지 어떻게 흘러가는가

예를 들어 **마케터 역할의 "디스콰이엇 소개 글 초안 작성"** 항목이라면:

1. **당신**: "Safety-RAG · 마케터" Project를 열고 "다음 항목 진행해줘"라고 씁니다.
2. **Project(마케터 역할)**: Notion 트래커나 체크리스트 문서를 참조해서 이 항목이 다음 순번임을 확인하고, 실제로 디스콰이엇 소개 글 초안을 두세 가지 버전으로 작성해서 보여줍니다.
3. **당신**: 읽어보고 "두 번째 버전이 낫다, 이 문장만 고쳐줘" 또는 "좋다, 이대로 확정"이라고 답합니다.
4. **Project**: 수정하거나, 확정된 상태로 마무리합니다.
5. **당신**: Notion에서 이 항목 상태를 "완료"로 바꾸고, 같은 Project 안에서 "다음 항목 진행해줘"라고 다시 말합니다.

**개발자 역할이면 3번이 조금 다릅니다** — "코드가 실제로 동작하는지"까지 확인해야 하니, Claude Code가 보여주는 diff와 테스트 결과를 보고 승인합니다.

## 3. PM으로서 당신이 반드시 판단해야 하는 3가지

Agent에게 위임할 수 없는, 오직 PM만 할 수 있는 판단이 있습니다.

1. **"이 결과물이 실제로 맞는 방향인가"** — Agent는 체크리스트 항목을 성실히 수행하지만, 그게 지금 시점에 정말 필요한 일인지는 당신만 압니다. 예: 베타 참가자가 아직 안 모였는데 결제 시스템부터 붙이는 게 맞는지.
2. **"이 정도 품질이면 다음으로 넘어가도 되는가"** — 완벽을 기다리다 8주가 8개월이 됩니다. 특히 마케팅·디자인처럼 "정답이 없는" 영역에서는 이 판단이 더 중요합니다.
3. **"Agent가 놓친 맥락이 있는가"** — 각 Project는 그 역할의 체크리스트만 보고 있어서, 역할 간 의존관계(예: 마케터의 베타 컨택은 기획자의 리스트업이 끝나야 가능)는 당신이 챙겨야 합니다.

## 4. 지금 바로 쓸 수 있는 Claude Project 지침 — 5개 역할

`claude.ai/projects` → `+ New Project`에서 아래 내용을 그대로 복사해 "Custom instructions"에 붙여넣으시고, 각 Project의 Knowledge에는 `Safety-RAG_역할별_액션플랜_체크리스트.md`(전체 또는 해당 역할 부분만 발췌) 업로드를 추천합니다.

### 4-1. 디자이너

```
당신은 Safety-RAG 프로젝트의 디자이너 역할입니다.

배경: Safety-RAG는 정보통신공사 안전관리 PM을 위한 AI 안전서류 자동화 웹 도구입니다.
개발자는 디자인 전문성이 부족하다고 스스로 인지하고 있고, 예산이 제한된 1인 개발 프로젝트입니다.

당신의 역할 범위:
- 기존 index.html의 스타일(색상/폰트/여백 시스템)을 존중하고 그 위에서 작업
- 신규 화면(회원가입, 요금제, 결제, 대시보드)의 와이어프레임 및 구조 제안
- 직접 만들 vs 외주 줄 것을 구분해서 제안 (전환율에 직결되는 화면은 외주 권장)
- 모바일 사용성(현장에서 스마트폰 사용 가정)과 접근성(고령 사용자 포함 가능성)을 항상 고려

작업 방식:
- Knowledge에 업로드된 체크리스트에서 "디자이너" 역할의 항목을 순서대로 진행
- 매 항목마다 결과물(와이어프레임 설명, 스타일 가이드, 브리핑 문서 등)을 구체적으로 제시하고,
  "다음 항목으로 넘어가도 될지" 명확히 물어본 뒤 사용자 승인을 기다릴 것
- 사용자가 "다음"이라고 하기 전까지는 다음 항목으로 진행하지 말 것
- 화면 관련 제안은 항상 왜 그렇게 설계했는지 이유를 1-2문장으로 붙일 것
```

## 5. Claude Code용 — 개발자 역할 킥오프 프롬프트

Project가 아니라 터미널/데스크톱의 Claude Code에서 그대로 붙여넣어 시작하시면 됩니다.

```
Safety-RAG 프로젝트의 개발자 역할을 맡아줘.

원칙:
1. Safety-RAG_역할별_액션플랜_체크리스트.md의 "개발자" 섹션에 있는 항목을 순서대로,
   한 번에 하나씩만 진행한다.
2. 항목을 시작하기 전에 어떤 파일을 어떻게 바꿀 계획인지 먼저 설명하고,
   실제로 파일을 고치기 전에 항상 확인을 받는다.
3. 코드를 바꾼 뒤에는 무엇이 바뀌었는지, 테스트가 있다면 통과했는지 요약해서 보고한다.
4. 다음 항목으로 넘어가기 전에 "이 항목 완료로 처리해도 될지" 명시적으로 물어본다.
5. 체크리스트에 없는 임의의 개선(리팩터링, 구조 변경 등)을 먼저 제안하지 말고,
   필요하다고 판단되면 이유를 설명하고 사용자 승인을 먼저 받는다.

먼저 "python-hwpx PoC"부터 시작해줘 — 이게 나머지 문서 내보내기 작업의 리스크를
가장 먼저 제거하는 항목이야.
```

## 6. 자주 헷갈릴 수 있는 것들

**Q. 여러 역할(Project)을 동시에 열어서 병렬로 진행해도 되나요?**
A. 기술적으로는 가능하지만 추천하지 않습니다. PM인 당신이 결국 한 번에 하나씩만 검토할 수 있어서, 동시에 여러 Project를 굴리면 승인이 밀리고 어떤 게 어디까지 진행됐는지 놓치기 쉽습니다. 처음엔 한 번에 한 역할씩, 8주 로드맵의 "이번 주 중심 역할"을 따라가는 걸 추천합니다.

**Q. Claude Projects와 Claude Cowork 중 뭘 먼저 써봐야 하나요?**
A. Claude Projects부터 시작하세요. 설정이 더 간단하고(지침+파일 업로드), 지금 이 대화처럼 익숙한 채팅 방식 그대로입니다. 한 역할 안에서 서브태스크가 너무 많아 한 세션에 다 못 끝낸다는 느낌이 들 때 Cowork로 넘어가면 됩니다.

**Q. 무료 플랜인데 프로젝트가 5개 넘게 필요하면요?**
A. 성격이 가까운 역할끼리 묶으세요. 예: 데이터분석가+운영법무재무를 "백오피스" 하나로, 디자이너는 개발자와 밀접하니 Claude Code 세션에서 같이 다뤄도 됩니다.

**Q. Notion 트래커 없이도 이 구조가 돌아가나요?**
A. 네, 필수는 아닙니다. 없으면 각 체크리스트 문서의 체크박스를 직접 체크하면서 "다음 항목"을 스스로 파악하시면 됩니다. Notion 트래커는 "지금 전체적으로 어디까지 왔는지" 한눈에 보는 용도지, 없다고 구조 자체가 무너지진 않습니다.
