# Safety-RAG 핸드오프: 성능 개선 + 표준작업계획서 남은 검토사항

## 현재 상태 요약
표준 작업계획서(굴착작업) 인용 정확성 이슈 3건(조번호 환각, 벌칙조항 과잉검열,
맨홀/밀폐공간 혼동)은 여러 라운드의 수정·재생성 테스트를 거쳐 모두 해결
확인됨. 이 문서는 그 이후 최신 생성 결과물을 재검토하며 새로 발견한 2건과,
생성 속도(현재 30초 이상 소요) 개선 방안을 다룬다.

---

## A. 생성 속도 개선

### 문제
문서 생성에 30초 이상 소요됨. 원인 진단: 이 도구는 문서 전체(표 뼈대,
서명란, 보일러플레이트 문구 포함)를 매 요청마다 처음부터 전부 생성하는
구조라 출력 토큰 자체가 매우 많음 (현재 굴착작업 초안 기준 17개 섹션,
다수의 표).

**중요한 전제**: Claude API의 prompt caching은 입력 처리 속도(첫 토큰까지의
시간)만 단축하며, 출력 토큰 생성 속도에는 영향을 주지 않음(공식 문서 명시
사실). 즉 지금처럼 출력 자체가 긴 구조에서는 caching만으로 "생성이 끝나는
시간"이 크게 줄지는 않음. 이 전제를 깔고 아래 세 가지를 우선순위대로
검토할 것.

### 1순위: 스트리밍 적용 (난이도 낮음, 체감 효과 큼)
- 현재 `index.html`은 스피너만 표시하고 전체 응답이 완성될 때까지 대기하는
  구조 (`generate-status`에 "최대 30초 정도 걸릴 수 있습니다" 문구 존재 —
  스트리밍 미적용 상태로 추정, 코드 확인 필요)
- Claude API 스트리밍 응답을 적용하면 실제 총 생성시간은 그대로여도 첫
  문장이 1~2초 내 화면에 나타나기 시작해 체감 대기시간이 크게 줄어듦
- 구현 난이도가 가장 낮고 효과가 즉각적이므로 우선 적용 권장

### 2순위: Prompt caching 적용 (난이도 중간, 비용·일부 지연 개선)
- 현재 work_type별 법정별표 전문을 매 요청마다 반복 전송하는 구조라
  caching의 전형적 적합 사례
- system 프롬프트 + KB 컨텍스트(work_type 매칭 섹션 포함) 끝에
  `cache_control: {"type": "ephemeral"}` 브레이크포인트 추가
- 캐시 적중 시 해당 구간 처리 속도 2배 이상 개선, 입력 토큰 비용 최대 90%
  절감 (공식 문서 기준)
- 최소 캐시 가능 길이(약 1024 토큰) 조건은 현재 KB 주입량 고려 시 충분히
  충족될 것으로 판단됨
- 주의: 위 A 서두의 전제대로, 이건 "입력 처리" 개선이지 "출력 생성"
  개선이 아님. 체감 개선은 스트리밍보다 작을 수 있음

### 3순위: 템플릿+AI 하이브리드 구조로 전환 (난이도 높음, 근본적 개선)
- 표 헤더, 서명란, "결재 순서: 담당자 작성 → 검토자 검토 →..." 같은
  고정 보일러플레이트 문구는 매번 AI가 처음부터 재생성할 필요가 없음
- 코드 레벨에서 고정 템플릿(표 뼈대, 서명란, 반복 문구)을 미리 구성해두고,
  AI는 현장별로 달라지는 내용(위험요인 서술, 안전대책, 작업순서, 매설물
  대책 등)만 채우도록 구조 변경
- 출력 토큰 자체가 줄어들어 실질적인 생성시간 단축 효과
- **부가 효과**: 아래 B-2(문서 구조 비일관성) 문제도 템플릿 고정으로 함께
  해결됨
- 리팩터링 범위가 크므로, 4개 작업유형 안정화가 완전히 끝난 뒤 별도
  스코프로 진행 권장. 지금 당장 착수하지 말 것

## B. 최신 굴착작업 초안 재검토 — 오픈 아이템 2건

### B-1. 위험성 점수 구간표 출처 불명확 — [조치 완료, 새 하위 버그는 C-1 참조]
최신 생성 결과물에 다음 각주가 포함됨:
> "위험성 점수 구간: 1~4 낮음 / 5~9 중간 / 10~25 높음"

이 구체적인 구간 값(1~4/5~9/10~25)은 당시 `위험성평가_실시규정.txt`에
명시되어 있지 않았음. **이후 사용자가 이 구간표를 `위험성평가_실시규정.txt`에
정식 반영 완료함** (FULL_INCLUDE_DOCUMENT_TYPES에 포함되어 위험성평가표
생성 시 항상 100% 주입되는 것까지 코드로 확인됨). 다만 KB 반영 *이전에*
생성된 초안에서 이 구간표가 물결표(~) 없이 깨진 형태("14", "59")로
출력되는 표기 오류가 발견됨 — 이는 KB 공백 문제가 아니라 별도의 표기
재현성 문제이므로 C-1로 분리하여 다룸.

### B-2. 문서 섹션 구조가 생성마다 달라짐
동일 조건(강남지사_광케이블, 굴착작업)으로 생성한 결과물을 3회 비교한 결과,
매번 섹션 순서·번호·소제목 배치가 달라짐 (예: "굴착공사 사전신고"가 어떤
회차는 사전조사 섹션의 하위 항목, 어떤 회차는 독립 섹션). 내용 정확도와는
무관한 별개 문제.

현장에서 동일 양식을 반복 사용/파일링해야 하는 문서 특성상 구조 일관성이
중요할 수 있음. 근본 해결책은 위 A-3(템플릿+AI 하이브리드)과 동일하므로,
지금 당장 별도 조치보다는 A-3 작업 시 함께 해결하는 것을 권장. 다만 A-3
착수 전까지 임시로 system_prompt에 "다음 섹션 순서를 반드시 따를 것"과
같은 고정 구조 지시를 추가하는 것도 단기 완화책으로 고려 가능.

## C. 신규 발견사항 — 위험성평가표(일산지사_국사장비반입) 진단 결과

배경: "낯선 작업 조합에서 프로젝트 정보를 제대로 반영 못 하는 것 아니냐"는
의심으로 근본 원인 진단을 진행함. 결론: 입력 파싱 자체는 정상 작동. 실제
원인은 (1) 이미 고친 문제의 표기 재발, (2) KB 커버리지가 얇은 영역에서의
순수 창작, (3) 커버리지 자체가 없는 work_type의 미승격 — 세 가지로 분리됨.

### C-1. 위험성 점수 구간표 물결표(~) 표기 재발
`generate_draft.py`의 system_prompt에 이미 "물결표를 절대 생략하지 마라"는
지시가 존재함(과거 동일 문제 패치 이력). 그런데도 이번 초안에서 "1~4"→"14",
"5~9"→"59"로 물결표가 누락됨(단, "10~25"는 정상 출력 — 3개 중 2개만 실패).

이건 KB 공백이 아니라 **이미 KB에 정확히 있는 값을 생성 과정에서 잘못
베낀 것**. 조번호 인용 환각이 프롬프트 보강 후에도 확률적으로(약 25%)
재발했던 것과 동일한 성격의 문제 — 프롬프트 지시만으로는 100% 방지가
안 된다는 이 프로젝트의 반복된 관찰과 일치함.

**권장 조치**: `find_unverified_citations()`와 같은 구조의 코드 레벨
체커를 숫자 범위 표기에도 추가. KB 원문에 있는 "1~4", "5~9", "10~25"
패턴이 최종 초안에서 물결표 없이("14", "59" 등 붙어쓴 형태로) 나타나면
탐지해서 경고를 붙이는 방식. 기존 인용 체커 코드를 재사용 가능할 것으로
보임.

### C-2. 활선설비 이격거리 "30cm" — 순수 창작 확정 [C-3 완료 후 재관찰 결과: 부분 개선, 미해소]
위험성평가표(일산지사_국사장비반입) No.5 항목에 "이격거리(최소 30cm 이상)
확보"라는 구체 수치가 등장. KB 7개 파일 전체를 대조했으나 이 수치의
근거를 어디에서도 찾을 수 없음 — 지금까지 발견된 것 중 가장 명확한 순수
창작 사례.

**권장 조치**: 이 특정 값 자체를 고치는 것보다, C-3(아래)의 work_type
승격이 이루어지면 관련 KB 섹션이 생기면서 자연 해소될 가능성이 높음.
다만 근본적으로는 "법조문 형식이 아닌 일반 수치"에 대한 방어가 여전히
약하다는 걸 보여주는 사례이므로, 아래 D(성능개선 완료 후 후속 과제)에
포함해 지속 관찰할 것.

**C-3 완료 후 재관찰(2026-07-12, 동일 project_info로 위험성평가표 1회
재생성)**: 원래 문제였던 "활선설비 이격거리"에는 더 이상 구체 수치가
안 붙고 "이격거리를 사전 실측하고 안전 여유 확보 여부 확인"으로 서술됨 —
그 자리는 개선. 그런데 같은 초안의 위험요인 6(기존 운용설비 손상)에서
"통로 폭과의 여유 공간(좌우 각 최소 30cm 이상)"이라는 **새로운 구체
수치가 다른 위험요인에 창작**되어 나타남. 검색된 근거 청크 5개에는
새로 승격한 "차량계 하역운반기계등" 섹션 자체가 포함되지 않았음(RAG
top-5에 안 잡힘) — 즉 이번 결과가 좋아 보인 부분을 C-3 덕분이라고
단정할 근거는 약함. 표본 1개라 통계적 결론은 낼 수 없지만, 적어도
"패턴이 완전히 해소됐다"는 아니고 **"법조문 형식이 아닌 일반 수치에
대한 가드레일 부재"라는 근본 진단은 여전히 유효**. 별도 API 비용 없이
여기서 관찰을 멈추기로 함(사용자 결정, 2026-07-12) — 추가 반복 검증이나
일반 수치 가드레일 설계는 후속 과제로 남김.

**후속 조치 완료(2026-07-13)**: [docs/superpowers/specs/2026-07-13-이격거리-수치-가드레일-design.md](../superpowers/specs/2026-07-13-이격거리-수치-가드레일-design.md)
설계에 따라 `find_unverified_clearance_values()` 추가 완료. 조문 인용
체커와 동일한 방향(초안에 있는데 참고자료에 없으면 미검증)으로 이격거리/
간격 계열 수치를 감시하며, system_prompt에도 동일 원칙의 가드레일 문구를
추가함. "일반 수치 가드레일 부재"라는 근본 진단에 대한 코드 레벨 대응이
이제 존재함(단, 스코프는 이격거리/간격으로 한정 — 인원수·무게 등 다른
수치는 여전히 스코프 밖).

### C-3. 차량계 하역운반기계(지게차) — work_type 미승격 상태로 실사용 발생
`common.py`의 `WORK_TYPE_SECTION_MARKERS`에 차량계 하역운반기계가
매핑되어 있지 않고, `표준작업계획서_법정별표.txt`에서도 "3. 향후 확장
후보"로만 존재함. 그런데 실사용 시나리오(국사 내 지게차를 이용한 장비
반입)가 이미 발생했고, 정식 커버리지가 없는 상태로 생성이 진행되며 C-2의
창작을 유발한 것으로 판단됨.

**권장 조치**: 차량계 하역운반기계를 정식 5번째 work_type으로 승격.
`표준작업계획서_법정별표.txt`에 이미 초안 내용(사전조사: 별도 규정 없음
/ 필수기재사항 2개: 위험 예방대책, 운행경로 및 작업방법)이 있으므로,
`WORK_TYPE_SECTION_MARKERS` 매핑과 프론트엔드 작업유형 선택 카드 추가가
핵심 작업. 기존 4개 work_type 구현 패턴을 그대로 재사용 가능.
