# PRD — 안전서류 AI 초안 생성 시스템 (safety-rag)

> Product Requirements Document
> 최종 업데이트: 2026-07-08

---

## 1. 개요

정보통신공사 현장에서 안전 관련 서류(위험성평가표, TBM 일지 등)를 준비·작성·검토하는 과정을
AI가 보조하는 도구. RAG(Retrieval-Augmented Generation) 구조로 법령/표준 서식을 근거 삼아
초안을 생성하며, 최종 검토·승인은 반드시 사람(안전관리자)이 수행하는 것을 전제로 한다.

## 2. 배경 및 계기

정보통신공사 프로젝트 PM 수행 중, 안전 관련 서류를 미리 준비하고 검토받는 데 시간이 오래
걸리고 무엇부터 해야 할지 판단하기 어려웠던 실제 경험에서 출발했다.

## 3. 프로젝트 목적 (중요)

**이 프로젝트는 "RAG 기술을 배우기 위한 사이드프로젝트/학습" 목적으로 진행한다.**
비즈니스(매출) 목적이 아니라, 본인이 먼저 사용(dogfooding)하며 검증하고 필요 시 확장하는
방향이다. 이 원칙은 코드 퀄리티, 시간 투자, 의사결정 기준 전반에 적용된다 — 완벽한 아키텍처나
상용화 수준의 안정성보다 RAG 파이프라인의 각 구성요소(청킹, 임베딩, 검색, 생성)를 직접
설계·실험해보는 것이 우선이다.

Phase 3(임베딩 모델 비교 실험)에서 이 원칙이 실제로 유효함을 확인했다 — "더 나은 임베딩
모델을 쓰면 검색이 개선될 것"이라는 가정보다, 직접 3개 모델을 비교 실험해 "지식베이스 문서
구조가 더 중요한 변수"라는 결론에 도달한 것 자체가 학습 목적 프로젝트로서 의미 있는 성과다.

## 4. 타깃 사용자

1차 사용자는 본인(20년 IT 경력, 정보통신공사 PM/안전관리자 업무 경험 보유). Phase 4에서
텔레그램 미니앱으로 전환하며 배포 URL과 봇 링크가 외부에 노출되므로, "나 혼자"가 아니라
**초대받은 소수(비공개 베타)** 로 범위를 정한다 — 완전 공개 시 API 비용이 사용자 수에
비례해 증가하고, 안전서류 도메인 특성상 검증 없는 공개는 오남용 리스크가 있기 때문
(Phase 5, `docs/PHASE5_SIGNUP_APPROVAL.md` 참고). 검증되면 비슷한 상황의 소규모
정보통신공사 현장 관계자로 초대 대상을 확장할 수 있다.

## 5. 문제 정의

- 안전 서류 종류가 많고 각 서류가 요구하는 형식·절차가 달라 처음 접하면 무엇부터 준비해야
  할지 파악이 어렵다
- 위험성평가와 TBM처럼 실제로는 연계되어야 하는 서류들이 현장에서 따로 작성되는 경우가 많다
- 50인 미만 소규모 사업장은 여전히 서류 위주 수작업 관리에 머물러 있다

## 6. 시장 조사 요약

- 경쟁 서비스 "안전해YOU"(safeyou365.com): 채팅형 입력으로 위험성평가표를 빠르게 생성,
  다국어 지원. 다만 일반 건설현장(철근/콘크리트) 중심으로 보이며 정보통신공사 특화 여부와
  TBM 연동 여부는 불확실
- 에스원, 포커스에이아이 등은 AI CCTV + 하드웨어 기반의 대기업/중견기업 대상 솔루션으로,
  1인개발자가 정면 승부할 영역이 아님
- 결론: 정보통신공사 특화 + 위험성평가-TBM 연동이라는 좁고 뾰족한 지점이 상대적으로 비어있음

## 7. 차별화 포인트

1. **정보통신공사 특화**: 통신주 작업, 광케이블 매설, 국사 내 작업 등 일반 건설과 다른
   위험요인을 지식베이스에 명시적으로 반영
2. **위험성평가 ↔ TBM 실제 연동**: TBM의 핵심 유해·위험요인은 원래 그날의 위험성평가표에서
   발췌하는 것이 원칙 — 이를 별도 문서가 아닌 하나의 워크플로우로 연결 (Phase 2에서 실제
   동작 확인 완료)

## 8. 법적 주의사항 (필수 준수)

- 이 도구는 법률 자문이나 법적 인증을 대체하지 않는다
- 모든 생성 결과 하단에 "이 초안은 참고용이며, 최종 검토 및 승인은 안전관리자가 직접
  수행해야 합니다" 문구를 반드시 포함한다
- 참고 자료에 없는 내용은 추측해서 생성하지 않는다 (시스템 프롬프트로 강제)

## 9. 핵심 기능 (현재 범위)

| 기능 | 상태 |
|---|---|
| 지식베이스 기반 문서 초안 생성 (위험성평가표, TBM 일지 등) | ✅ 구현 완료 |
| 위험성평가 → TBM 결과 자동 연동 (현장명 기준) | ✅ 구현 완료 |
| 검색 품질 단독 테스트 도구 (다중 모델 지원) | ✅ 구현 완료 |
| 다중 임베딩 모델 비교 실험 (Voyage / Cohere / KURE-v1) | ✅ 완료 (Phase 3) |
| 문서종류 메뉴 선택, 다중 위험성평가 회차 선택 (개인용 실사용성 개선) | ✅ 완료 (Phase 3.5) |
| 웹 인터페이스 (FastAPI 백엔드 + 텔레그램 미니앱 프론트엔드), Railway 배포 | ✅ 완료, 실기기 검증 완료 (Phase 4) |
| 가입 승인 플로우(봇 인라인 버튼 승인/거절), 사용자별 데이터 격리, 일일 사용량 제한, Railway Volume 영구 저장소 | 🟡 설계 완료, 코드 적용 전 (Phase 5) |

## 10. 비목표 (Non-goals, 현재 단계 기준)

- 법적 인증/자동 승인 기능은 만들지 않는다
- 다수 사용자를 위한 인증/계정 시스템은 현재 범위 밖이다 (Phase 4에서 재검토)
- 위험성평가 점수를 자동으로 산정·확정하는 것은 목표가 아니다 (초안 생성까지가 범위이며
  최종 판단은 사람이 한다)

## 11. 성공 기준 (학습 목적 프로젝트 기준)

- 본인이 실제 현장 시나리오에서 이 도구로 생성한 초안을 실사용 가능한 수준의 참고자료로
  느끼는가 (dogfooding 검증) — ✅ Phase 2에서 위험성평가·TBM 생성 결과로 확인
- RAG 파이프라인의 각 구성요소(청킹 전략, 임베딩 모델, 검색 정확도)에 대해 최소 1회 이상
  비교 실험을 수행하고 결과를 기록했는가 — ✅ Phase 3에서 3개 임베딩 모델 비교 완료
- 검색이 특정 문서 유형에서 체계적으로 실패하는 케이스(예: 관리비 문서 누락)를 식별하고
  원인 가설을 세웠는가 — ✅ 식별 및 원인 규명(청킹 구조 문제), 실제 개선까지 검증 완료

## 12. 참고 원칙 (1인개발 10원칙 중 이 프로젝트에 적용되는 항목)

- 원칙 1 (목적 명확화): 사이드프로젝트/학습으로 확정 → 완료
- 원칙 3 (유틸 서비스부터): 현재 범위는 개인용 CLI 도구로, 콜드스타트 문제와 무관 → 해당 없음
- 원칙 4 (빠른 MVP): Phase 1 MVP를 최소 기능으로 빠르게 구현하고 본인 검증부터 시작 → 완료
- 원칙 7 (분석 도구 조기 장착의 정신): 검색 결과를 눈으로 확인하는 디버그 도구를 초기부터
  마련 → 완료 (`test_search.py`, Phase 3에서 다중 모델 비교까지 확장)
- 그 외 마케팅/리텐션/커뮤니티 관련 원칙은 Phase 4(타인 사용 확장) 이후에 적용 대상이 된다
