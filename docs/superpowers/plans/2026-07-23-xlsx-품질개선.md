# XLSX 결과물 품질 개선 (현장명·점수·용어) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 사용자가 실제 XLSX 결과물을 검토하며 발견한 3가지 품질 문제를 고친다 — (1) 현장명에 언더스코어가 그대로 노출됨, (2) 위험성/가능성/중대성 점수 셀이 숫자가 아니라 텍스트라 엑셀에서 계산이 안 됨, (3) 표준작업계획서 "담당" 칸의 "전원"이라는 표기가 "전원(電源)"과 헷갈림.

**배경**: XLSX는 `generate_draft.py`가 만든 Markdown `draft` 텍스트를 `export_xlsx.py`가 그대로 파싱해 셀에 옮기는 구조라([2026-07-22-xlsx-내보내기.md] 참고), 원본 Markdown 자체에 있는 문제는 세 문서(위험성평가표/표준작업계획서/TBM일지)에 전부 그대로 나타난다. 세 문제의 원인 위치가 서로 다르다:
- (1) 현장명: `_build_generation_prompt()`가 `project_name`을 가공 없이 프롬프트에 그대로 넣어서(`generate_draft.py:255`), LLM이 그 값을 문서에 그대로 옮겨 적음.
- (2) 점수 텍스트: system_prompt가 "점수를 적을 때마다 반드시 그 옆에 '(AI 제안값, 현장 확인 필수)'라고 표시해"라고 명시적으로 지시하고 있어(`generate_draft.py:271-272`), 이건 안전상 의도된 동작이다 — 숫자 옆에 "AI가 제안한 값이니 현장에서 재확인하라"는 경고를 반드시 붙이라는 요구사항. 이 경고 자체를 없애면 안 되고, **셀에 숫자와 경고문이 하나로 뒤섞여 들어가는 것**만 문제다.
- (3) "담당" 표기: 지식베이스(`knowledge_base/`)를 찾아봐도 "담당: 전원"이라는 문구의 출처가 없다 — system_prompt가 "작업 단계별 절차" 섹션의 표 구성을 지시하긴 하지만 열 이름·용어를 구체적으로 못 박지 않아서, LLM이 "모든 인원"이라는 뜻으로 "전원"을 쓰면서 같은 문서 안의 "전원(電源) 인입 작업" 같은 전기 용어와 겹쳐 보이는 것으로 판단된다.

**Architecture:**
- (1)은 **결정론적 Python 문자열 치환**으로 고친다 — LLM에게 "언더스코어를 공백으로 바꿔써"라고 지시하는 것보다, 애초에 프롬프트에 넣기 전에 코드로 바꿔버리는 게 100% 확실하다. `project_name` 자체(파일 조회 키)는 그대로 두고, 프롬프트에 노출하는 표시용 값만 별도 함수로 가공한다.
- (2)는 **XLSX 변환 레이어(`export_xlsx.py`)에서만** 고친다 — system_prompt를 바꿔 표 구조 자체를 바꾸는 방법도 있지만, 그러면 웹앱 화면 표시·다른 문서유형까지 전부 영향을 받고 프롬프트 회귀 테스트가 필요해진다. 대신 셀 값이 `"15(AI 제안값, 현장 확인 필수)"` 패턴과 일치하면, 셀에는 순수 숫자 `15`만 넣고 안내 문구는 **엑셀 셀 메모(comment)**로 옮긴다 — 숫자는 합계·정렬·필터가 가능해지고, 경고 문구는 셀에 마우스를 올리면 그대로 보여서 사라지지 않는다.
- (3)은 **system_prompt에 짧은 용어 구분 지침을 추가**하는 LLM 프롬프트 수정이다. 코드로 안전하게 치환할 수 없는 이유: 같은 문서 안에 "전원(電源) 재투입 절차"처럼 "전원"이 올바르게 쓰인 곳도 있어서, 무작정 문자열 치환하면 그 부분까지 잘못 바뀐다. 반드시 의미를 구분해야 하므로 LLM 지시로 처리한다.

## Global Constraints

- 저장소의 `test_*.py`는 `python test_X.py`로 직접 실행하는 PASS/FAIL 프린트 스크립트 관례를 따른다 (pytest 아님).
- `generate_draft.py`를 변경하면 `.git/hooks/pre-commit`이 `python test_worktype_citations.py` 실행을 권장하는 안내를 띄운다 — 이 테스트는 **실제 Claude API를 호출해 비용이 발생**하므로, 자동으로 실행하지 말고 반드시 사용자에게 먼저 물어본 뒤 승인 시에만 실행한다 (저장된 API 토큰으로 외부 서비스를 호출하기 전에 확인받는다는 기존 원칙).
- `project_name`(파일 조회 키, `PROJECTS_DIR/{user_id}/{project_name}.json`)과 "표시용 현장명"(문서 본문에 보이는 값)은 반드시 분리한다 — 조회 키 쪽은 언더스코어를 유지해야 기존에 저장된 레코드 조회가 깨지지 않는다.
