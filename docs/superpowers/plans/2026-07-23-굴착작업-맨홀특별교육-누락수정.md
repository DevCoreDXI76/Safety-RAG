# 굴착작업 표준작업계획서 "맨홀작업 특별교육(§34)" 언급 누락 수정 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 굴착작업 표준작업계획서를 생성할 때 "맨홀작업 특별교육(제34호)"이 산업안전보건법 시행규칙 별표5의 밀폐공간(제35호)과 혼동되지 않고 안정적으로 정확히 언급되도록 한다.

**배경 조사 결과 (2026-07-23)**:
- `test_worktype_citations.py`로 굴착작업 표준작업계획서를 9회(최초 1회 + Task3 이후 4회 반복 + Task3 이전 4회 반복 대조실험) 생성해본 결과, "맨홀작업"/"34호" 언급이 **9회 중 9회 모두 누락**됐다. Task3(2026-07-23 "전원"→"전체 인원" 프롬프트 변경)이 원인이 아님을 대조실험으로 확인했다 — Task3 이전 커밋에서도 동일하게 4/4 누락.
- 근본 원인은 **두 지식베이스 파일의 역할 분리 문제**다:
  - `knowledge_base/표준작업계획서_법정별표.txt`(굴착작업 섹션)는 **직접 주입**되는(RAG 검색 대상이 아닌, 작업유형 선택 시 항상 전체 포함) 파일이고, 여기엔 이미 "광케이블 지중매설, **맨홀/핸드홀 신설**을 위한 굴착 작업에 해당"(현재 66번째 줄)이라고 명시돼 있다 — 즉 이 굴착작업이 맨홀 관련 작업이라는 사실 자체는 매번 확실히 프롬프트에 들어간다.
  - 반면 "맨홀작업(제34호)"과 "밀폐공간(제35호)"을 구분하는 내용은 `knowledge_base/안전보건교육_법정시간및내용.txt`에만 있고, 이 파일은 **RAG 유사도 검색으로만** 컨텍스트에 들어간다. 검색 자체는 안정적으로 상위 5개 안에 들어오지만(9회 모두 동일한 검색결과 확인), 이 내용은 원래 "안전보건교육일지" 문서유형을 위한 것이라 "표준 작업계획서"의 필수 섹션 구성(system_prompt가 지시하는 1~9번 섹션)에는 대응하는 자리가 없다 — 그래서 모델이 검색된 내용을 봐도 굳이 표준작업계획서 본문에 옮겨 적을 유인이 약하다.
  - 즉 "맨홀/핸드홀 작업이라는 사실"은 확실히 전달되지만, "그러니 제34호를 근거로 인용해야 하고 제35호와 혼동하면 안 된다"는 지시는 어디에도 없다.

**Architecture:** 시스템 프롬프트(`generate_draft.py`)가 아니라 **`표준작업계획서_법정별표.txt`의 굴착작업 섹션에 직접** 제34호/제35호 구분 문장을 추가한다. 이유:
1. 이 파일은 이미 굴착작업 선택 시 **매번 보장되어 직접 주입**되므로(RAG 검색 의존 없이), 다른 이미 안정적으로 등장하는 사실들(굴착공사정보지원센터, 제345~347조 등)과 동일한 신뢰 수준을 얻는다.
2. 굴착작업에만 해당하는 내용이라, 모든 문서유형에 영향을 주는 거대한 `system_prompt` 문자열에 지침을 추가하는 것보다 범위가 정확하다(다른 작업유형·다른 문서유형에 영향 없음).
3. 파일 자체의 "정보통신공사 적용 맥락" 문단과 같은 톤·형식(법령명+조번호+괄호 설명)을 그대로 따르면 되므로 프롬프트 엔지니어링 부담이 적다.

## Global Constraints

- `knowledge_base/*.txt`를 변경하면 `.git/hooks/pre-commit`이 `embeddings_voyage.json`도 함께 갱신됐는지 확인하고, 갱신 안 됐으면 **커밋을 막는다**(하드 블록). 반드시 `python build_knowledge_base.py --model voyage`를 먼저 실행해야 한다.
- `build_knowledge_base.py --model voyage`는 **Voyage AI 임베딩 API를 실제로 호출**한다(저장소 전체 청크를 재임베딩, 비용 발생). 저장소의 기존 원칙(저장된 API 토큰으로 외부 서비스 호출 전 확인)에 따라, 이 단계는 사용자에게 먼저 실행 여부를 물어본 뒤 승인 시에만 진행한다.
- 이 파일은 "직접 주입용"이라고 파일 1번째 줄에 명시돼 있으므로("RAG 유사도 검색 대상이 아니라... 프롬프트에 직접 전체 주입"), 새로 추가하는 문장도 굴착작업 섹션 안에 넣어야 한다 — 파일 밖(예: `안전보건교육_법정시간및내용.txt`)에 추가하면 다시 RAG 의존적이 되어 이번 수정의 의미가 없어진다.
- 실제 효과 검증(`test_worktype_citations.py --repeat`)은 유료 API 호출이므로, 이 역시 실행 전 별도로 사용자 승인을 받는다.

## Self-Review 메모

- **범위**: 이 수정은 굴착작업 표준작업계획서에만 영향을 준다 — 다른 4개 작업유형이나 다른 문서유형(위험성평가표, TBM일지 등)에는 영향이 없다. `system_prompt`(모든 문서유형 공통)가 아니라 굴착작업 전용 KB 섹션만 건드렸기 때문이다.
- **왜 시스템 프롬프트가 아니라 KB 파일인가**: 이미 이 세션에서 두 번(2026-07-22 XLSX 계획, 2026-07-23 품질개선 계획 Task 3) `system_prompt`에 지침을 추가했다 — 계속 추가하면 이미 거대한 프롬프트가 더 비대해지고, 모든 문서유형·모든 작업유형에 영향을 주는 변경이 늘어난다. 이번처럼 특정 작업유형에만 해당하는 사실은 해당 작업유형의 직접주입 KB 섹션에 넣는 것이 정확한 위치다.
- **완전한 해결은 아닐 수 있음**: LLM 특성상 직접주입 + 명시적 지침을 추가해도 100% 결정론적 등장을 보장하지는 못한다(다른 유사 사실들도 9회 중 1회 정도는 빠진 전례가 있음). 이 플랜은 "9/9 누락"이라는 명백한 결함을 "다른 안정적 사실들과 비슷한 수준"으로 끌어올리는 것을 목표로 하며, Step 5의 검증 결과가 기대에 못 미치면 추가 조치(예: system_prompt에 보강 지침 추가)가 필요할 수 있다.
