## 1인개발 10원칙 기준 현황 정리

### 1. 목적 (원칙 1 기준)

**사이드프로젝트/학습 목적**으로 이미 명확히 정해져 있다. 매출을 만드는 비즈니스가 아니라,
**RAG(Retrieval-Augmented Generation) 기술을 직접 설계하고 실험해보며 역량을 키우는 것**이
1차 목표다. 다만 학습용이라고 아무렇게나 만드는 게 아니라, 본인이 정보통신공사 PM 경험에서
실제로 느꼈던 문제(안전서류 준비의 번거로움)를 소재로 삼아서 **본인이 실제로 써볼 만한 수준
까지는 만들어보는 것**이 검증 기준이다.

이 목적 설정이 나머지 모든 결정에 영향을 준다 — 완벽한 아키텍처나 상용화 수준의 안정성보다,
"청킹을 이렇게 바꾸면 검색이 어떻게 달라지는가", "임베딩 모델을 바꾸면 결과가 어떻게
달라지는가" 같은 **실험과 이해**가 우선이다. 실제로 Phase 3에서 이 원칙대로 진행한 결과,
"모델을 바꾸면 다 해결될 것"이라는 가정을 실험으로 직접 반증하고 청킹 구조가 더 중요하다는
것을 확인했다 (자세한 내용은 아래 Phase 3 섹션 참조).

### 2. 무엇을 만들고 싶은가

정보통신공사 현장의 안전서류(위험성평가표, TBM 일지 등) 초안을 AI가 근거 자료 기반으로
생성해주는 CLI 도구. 핵심 차별점은 두 가지다:

- **정보통신공사 특화**: 일반 건설현장이 아니라 통신주 작업, 광케이블 매설, 맨홀 작업 같은
  이 업계 특유의 위험요인을 지식베이스에 반영
- **위험성평가 ↔ TBM 연동**: 원래 현장에서는 그날의 위험성평가 내용이 TBM에 반영되어야
  하는데, 이 워크플로우를 실제로 코드로 구현 (이미 동작 확인함)

법적 자문을 대체하는 게 아니라 어디까지나 "초안 생성 도구"이고, 최종 판단은 사람이 한다는
원칙(모든 생성물에 고지 문구 강제 포함)도 이미 지키고 있다.

### 3. 지금 Plan은 어디까지 왔나

| Phase | 내용 | 상태 |
|---|---|---|
| Phase 1 | 청킹/검색/생성 파이프라인 구축 (원칙 4: 빠른 MVP) | ✅ 완료 |
| Phase 2 | 본인 검증 — 위험성평가·TBM 연동 확인, 지식베이스 2종→5종 확장, 검색 품질 테스트 | ✅ 완료 |
| Phase 3 | 임베딩 모델 비교 실험 (Voyage vs Cohere vs KURE-v1) | ✅ 완료 |
| Phase 4 | 웹/봇 인터페이스로 확장 | ⏳ 착수 여부 결정 대기 |

지금은 **본인 혼자 CLI로 검증하는 단계**라, 원칙 6(마케팅), 7(유저 이벤트 분석 SDK),
8(리텐션), 9(커뮤니티), 10(1인사업 마인드) 같은 항목들은 아직 적용할 시점이 아니다 —
이건 Phase 4에서 실제로 남에게 서비스를 열 때 다시 꺼내볼 원칙들이다. 지금 단계에서 실제로
관련 있는 원칙은 1(목적 확정, 완료), 3(유틸 서비스로 시작, 이 도구는 개인용 CLI라
콜드스타트 문제 자체가 해당 없음), 4(빠른 MVP, 완료), 7(분석/디버그 도구 조기 장착 —
`test_search.py`로 검색 품질을 눈으로 확인하는 습관을 이미 실천 중)다.

## Phase 3 — 완료 내역

### 목표
동일한 테스트 케이스 세트로 임베딩 모델 간 검색 품질을 비교하고, Phase 2에서 발견한 "관리비
문서 누락" 문제가 모델에 따라 개선되는지 확인한다.

### 진행한 작업

- [x] `common.py`를 모델별 분기 구조로 리팩터링 (`embed_with_voyage()`, `embed_with_cohere()`,
      `embed_with_kure()` + `embed_texts()` 라우팅 함수)
- [x] 임베딩 결과를 모델별로 분리 저장 (`embeddings_voyage.json`, `embeddings_cohere.json`,
      `embeddings_kure.json`) — 비교를 위해 서로 덮어쓰지 않도록 설계
- [x] `build_knowledge_base.py`, `test_search.py`에 `--model` 인자 추가
- [x] Cohere API 키 발급, `embed-multilingual-v3.0` 모델 통합
- [x] KURE-v1(`nlpai-lab/KURE-v1`, sentence-transformers 로컬 추론) 통합, 지연 로딩 적용
- [x] 3개 테스트 케이스로 Voyage / Cohere / KURE-v1 3파전 비교 수행

### 1차 비교 결과 (문서 개선 전)

| 질의 | Voyage-3 | Cohere multilingual-v3 | KURE-v1 |
|---|---|---|---|
| 안전 전반 준비사항 (광범위) | 5개 문서 모두 등장 ✅ | 5개 문서 모두 등장 ✅ | 5개 문서 모두 등장 ✅ |
| 굴착 작업 서류 (경계 케이스) | 관리비 포함 (근소) | 관리비 누락 | 관리비 누락 |
| 교육 계획 | 교육 top-1,2 ✅ | 교육 top-1 ✅ | 교육 top-1,2 ✅ |

→ 세 모델 모두 사실상 동등한 품질. "관리비 문서 누락"이 모델과 무관하게 반복 발생 →
**임베딩 모델이 아니라 문서 구조(청킹 단위) 문제**라는 가설 수립.

### 근본 원인 진단 및 수정

- `산업안전보건관리비_가이드.txt`의 "정보통신공사 현장 특화 사용 예시" 섹션이 4개 항목을
  하나의 문단(하이픈 나열)으로 묶고 있어, "굴착"이라는 핵심 키워드의 임베딩 신호가 다른
  3개 항목(가스측정기, 안전대, 환기장비)에 희석됨
- 항목을 각각 독립된 문단으로 분리하고, 굴착 관련 항목은 구체적 문장으로 확장하여 키워드
  반복 노출

### 수정 후 재검증 결과 (3개 모델 모두 재빌드 후)

| 질의 | Voyage-3 | Cohere | KURE-v1 |
|---|---|---|---|
| 굴착 작업 서류 | **관리비 top-1** ✅ | **관리비 top-1** ✅ | **관리비 top-1** ✅ |

→ 세 모델 전부 즉시 개선 확인. 가설 검증 완료.

### 최종 결론

1. **임베딩 모델 선택**: 이 지식베이스 규모(5개 문서)에서는 Voyage-3 / Cohere
   multilingual-v3 / KURE-v1 간 검색 품질 차이가 유의미하지 않음
2. **모델 결정**: **Voyage-3를 그대로 프로덕션(`generate_draft.py`) 모델로 유지.**
   추가 API 의존성(Cohere)이나 로컬 호스팅 부담(KURE-v1)을 감수할 실익이 없음
3. **핵심 학습 성과**: 검색 품질 개선의 실제 레버리지는 임베딩 모델 교체가 아니라
   **지식베이스 문서의 청킹 단위 설계**에 있다는 것을 실험으로 검증. 앞으로 지식베이스에
   문서를 추가할 때 "핵심 키워드가 담긴 항목은 독립 문단으로 분리"하는 원칙을 유지한다
   (구체적 원칙은 `SPEC.md` 참조).
4. Contextual Retrieval, Reranking 등 추가 고도화 실험은 현재 규모에서는 우선순위가
   낮다고 판단해 보류 (지식베이스가 더 커지고 경계 케이스가 재발할 때 재검토)
